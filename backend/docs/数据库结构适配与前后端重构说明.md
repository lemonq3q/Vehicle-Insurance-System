# 数据库结构适配与前后端重构说明

## 1. 重构目标

本次重构保持原有页面、路由、接口响应外壳和主要操作流程不变，将代码的数据来源切换到 `insurance_saas` 的规范化结构。代码改动仅位于 `frontend` 与 `backend`。

## 2. 重构前后对比

| 领域 | 重构前 | 重构后 |
| --- | --- | --- |
| 商户分类 | `merchant.type` 保存中文并在 SQL/Java 中写死 | `biz_merchant.category_id` 关联 `biz_merchant_category`，代码使用稳定分类编码 |
| 商户人员 | 与系统用户共用 `user`、`user_role` | 仅下游商户使用 `biz_merchant_staff` + `biz_merchant_staff_role`；上游联系人直接保存在商户表 |
| 系统用户 | 通过角色名称从同一张用户表筛选 | `tenant_user` + `tenant_member`，登录身份与商户人员彻底分离 |
| 人员角色 | 联系人、收款人同时进入权限角色体系 | `CONTACT/CLERK/PAYEE` 不参与 Spring Security 权限判断 |
| 银行信息 | 代码中容易被误解为收款人字段 | 始终写入 `biz_merchant.bank/bank_card_num`，工单只保存商户银行信息快照 |
| 工单 | 一张宽表包含报价、费用、支付、核保、物流 | 主表与 5 组明细表事务写入，并直接聚合查询新表 |
| 起保时间 | 旧宽表独立字段 | 保留在工单主表的独立 BIGINT 字段，继续使用 Unix 秒时间戳 |
| 续保提醒 | 仅按起保时间判断一次且处理后不再展示 | 按工单创建日每年循环计算周年日前配置窗口；跟进状态记录所属周期，周期结束后由每日任务重置为未处理；永久停醒独立存储，测试提醒天数当前为 365 天 |
| 前端加载 | 所有路由组件首屏同步打包 | 路由级动态导入，页面按需加载 |
| 分页代码 | 多个列表复制相同 `el-pagination` | 统一为 `components/common/AppPagination.vue` |

## 3. 后端结构

### 3.1 common

- 增加 `MerchantCategoryCode`，集中维护新分类编码与旧展示名称的兼容映射。
- 增加 `MerchantStaff`、`MerchantStaffRole` 和业务角色目录。
- 原实体通过 `@TableName`、`@TableField` 对齐新表名和新列名。
- `EpochSecondsTypeHandler` 负责接口秒级时间戳与 MySQL `datetime` 的双向转换。

### 3.2 insruance

- `MerchantMapper.xml` 通过分类表的 `direction` 区分上下游，不再出现商户类型集合硬编码。
- 新增独立的 `/merchant-staff` 业务入口及 Service/Mapper；联系人唯一性和默认收款人由业务代码与数据库唯一约束共同保证。
- 系统用户查询、注册、审核和登录改用 `tenant_user`、`tenant_member`、`auth_role`、`auth_permission`。
- 工单通过 `WorkorderAggregateMapper` 在一个事务内写入主表、报价、上下游费用、支付、核保和物流。
- `WorkorderMapper.xml` 直接关联 `biz_workorder`、报价、费用、支付、核保、物流、车辆、商户和用户表，聚合为现有 `WorkorderDTO`；数据库中不创建旧表同名视图。
- 所有接口时间字段统一保持旧前端使用的秒级时间戳。迁移使用 `FROM_UNIXTIME(旧值)`，不再误按毫秒除以 1000。

### 3.3 saas

本次没有添加 SaaS 业务逻辑，继续保持空业务模块，只依赖 `common`，符合此前模块边界。

## 4. 前端结构

- 新增 `src/api/merchantStaff.js`，商户人员页面不再调用系统用户 API。
- `DownstreamUser.vue` 与 `EditDownstreamUser.vue` 使用独立商户人员接口及角色目录。
- `AppPagination.vue` 被上下游商户、商户人员和系统用户列表复用。
- `router/index.js` 改为路由懒加载。路径、权限守卫及页面行为不变。
- 本次没有引入新的 mock：项目已有页面直接使用真实 API，且本次保留了原响应结构 `{code,msg,data}` 与分页 `{total,table}`。

## 5. 数据迁移

迁移文件按以下顺序执行：

1. `backend/db/migration/V20260714_00_precheck.sql`（只读）
2. `backend/db/migration/V20260714_01_migrate_insurance_to_saas.sql`
3. `backend/db/migration/V20260714_03_verify_migration.sql`

迁移规则：

- 建立一个编号为 `LEGACY_INSURANCE` 的默认企业，承接旧系统数据。
- `admin`、`出单员` 迁移为租户系统用户；仅下游商户的联系人、店员、收款人迁移为商户人员，上游联系人保留在商户主表。
- 旧库商户 17、18 存在多个联系人。每个商户保留最早人员为 `CONTACT`，其余降为 `CLERK`，避免违反“一商户一个联系人”。
- 商户 17 存在多个收款人，全部保留为 `PAYEE`，最早人员设为默认收款人。
- 商户 7981 的旧分类为空，按下游车商店铺迁移；该记录名称为“批量插入临时测试”。
- 旧库存在 3 组重复商户编码。迁移时仅对重复编码追加 `_旧ID`，保证新库唯一约束可通过；商户主键及工单关联不变。
- 银行和银行卡字段从旧 `merchant` 迁移到 `biz_merchant`，工单支付表保存快照。
- `options_json`、`deductible_options_json` 保存选项集合，继续使用 JSON；默认值和工单已选值只是单个字符串，已改为 `varchar` 并清除迁移产生的 JSON 外层引号。
- 旧工单 `create_by` 实际全部指向商户联系人，迁移到 `source_staff_id`；`created_by/handle_by/updated_by` 只保留能够关联 `tenant_user` 的真实系统用户。

迁移已于 2026-07-15 通过 `mysql_ops` 执行。执行前确认全部目标业务表精确计数为 0；分类表保留 4 条种子数据。

迁移结果：

- 商户 10067 条、系统用户 5 条、旧用户拆分商户人员 12 条，源目标计数一致。
- 工单主表 20 条；报价、支付、核保、物流各 20 条，上下游费用各 20 条。
- 工单险种 460 条、工单附件 47 条、系统文件 99 条；续保时间直接迁移至工单主表。
- 行驶证 15 条、发票 5 条、合格证 5 条。
- 重复明细、重复联系人、重复默认收款人和悬空关联检查均为 0。
- 工单及支付历史时间、JSON 业务值、人员归属检查均与源数据一致。
- 2026-07-15 将四个单值选项字段由 JSON 改为 `varchar`，清洗默认选项 23 条、工单选项 460 条；变更前数据保存在 `biz_insurance_product_bak_20260715_scalar` 和 `biz_workorder_insurance_bak_20260715_scalar`。

## 6. 接口边界

- 前端现有 URL、查询参数、页面路由与响应字段保持不变；商户人员页面内部切换到了更明确的新 URL。
- 默认企业 ID 为 1，用于承接旧单租户业务。未来 SaaS 模块接入后，应从登录上下文解析 `enterprise_id`，不要继续写死。
- 仅保留前端接口字段兼容，不保留数据库旧表兼容层；后端查询和写入都以规范化表为唯一数据来源。

## 7. 验证

- 后端：`mvn.cmd -q -DskipTests package`
- 前端：`npm.cmd run build`
- 数据库：执行迁移后运行 `V20260714_03_verify_migration.sql`，计数必须一致且唯一性检查返回空集。

前端构建仍有既有的大资源告警（登录背景图和 vendor 包），不影响构建成功。
